So, by changing the userID to the account B‚Äôs userID, I could update the account B‚Äôs first and last name, grab it‚Äôs mail, referral code, companyID, subscription plan, number of employees, phone number, the company‚Äôs VAT identification number etc., while being authorized as account A. With the referral code, a user could use it to gain $30 as a ‚Äúgift‚Äù for getting referred by another user.

Probably the server did not identifiy who was sending the Authorization header and it was just making sure that a valid Authorization header existed.

While that‚Äôs all good, the userID of each account remains secret. There had to be some kind of MiTM attack to capture it and make a targeted attack. So I thought maybe I could generalize the attack, instead of targeting a single account.
By using dummy userIDs, in order to test the brute-force protection, I found out that the endpoint was vulnerable to brute-force attacks. Since the userID is following a certain pattern (a string of 24 hex characters), one could generate all the possible IDs and save them into a file with this python script:

import itertools

string = '0123456789abcdef'
file = open('userIDs.txt', 'w')
for p in itertools.product(string, repeat=24):
	writing = ''.join(p) + '\n'
	file.write(writing)
file.close()

With enough computing power, an attacker could change the first/last name of all users and grab their account and company info.
