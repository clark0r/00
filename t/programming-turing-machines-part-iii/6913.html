Procedure standards

As I mentioned in Part I, I use ‚Äò>‚Äô to mark the beginning of the tape. Some of my procedures perform an operation on two words on the same tape; I use a ‚Äò:‚Äô to mark the end of the first word. Any procedure that requires marking symbols to keep track of previously visited cells uses ‚Äò*‚Äô as the marker. As an example, I wrote a procedure that will reverse a bitstring:

; go to end of string
s1 > > r s1
s1 0 0 r s1
s1 1 1 r s1
s1 _ : l s2

; grab value to copy
s2 * * l s2
s2 0 * r s3
s2 1 * r s4
s2 > > r s5

; go to end of string on 0
s3 * * r s3
s3 0 0 r s3
s3 1 1 r s3
s3 : : r s3
s3 _ 0 l s6

; go to end of string on 1
s4 * * r s4
s4 0 0 r s4
s4 1 1 r s4
s4 : : r s4
s4 _ 1 l s6

; move left to :
s6 0 0 l s6
s6 1 1 l s6
s6 : : l s2

; delete *
s5 * l s5
s5 : l s5
s5 > > r s5

This procedure will first append a ‚Äò:‚Äô to the end of the tape. Then it will loop back and forth marking and copying characters in reverse order. Chaining

It‚Äôs time to move on to bigger and better things. Let‚Äôs revisit a procedure from the previous post:

s1 > > r s1
s1 0 1 r s1
s1 1 0 r s1
s1 _ _ l s2

s2 0 1 l s3
s2 1 0 l s2
s2 > > r s3

s3 0 0 l s3
s3 1 1 l s3
s3 > > r s4

s4 0 0 r s4
s4 1 1 r s4
s4 _ l s5

This procedure takes a bitstring as input and outputs two‚Äôs complement. This procedure isn‚Äôt terribly useful on its own, but when combined with a procedure to add two numbers, we get binary subtraction. Here‚Äôs an example of the one I made; it‚Äôs a bit lengthy so I won‚Äôt paste it here.
Looking at that, you can see how these procedures comply with the standard I outlined earlier: the add procedure expects two operands (of equal length) separated by a ‚Äò:‚Äô with no additional characters from Œì. Our two‚Äôs complement procedure expects one operand with no additional characters from Œì.
To combine these procedures into something useful, we need a way of storing and manipulating input tapes before they are passed to the next Turing machine. Let‚Äôs say we want to perform 2 - 1; based on what we know the procedures expect, we can start constructing our inputs:

2 = 10
1 = 01

So our tape for two‚Äôs complement will start out as follows:

>01
^

After the procedure runs, we will have:

>11
^

Next, we append the 2; since we‚Äôre adding now, order is irrelevant:

>11:10
^

Our add procedure will preserve bitness, so we don‚Äôt care about overflow. If we pass this modified tape into our add procedure, it will output the following after halting:

>01
^

And just like that, we have our answer.

More power

Like most of you, when I got to this point of the project, I began wondering ‚Äújust how far can I take this?‚Äù Pretty far. I wont paste the code I wrote, but I made a separate Chain() class for Turing machines. It contains the following components:

  1. An ordered list of procedures to run,
  2. A string to contain the tape passed immediately to a Turing machine,
  3. Four ‚Äúregisters‚Äù that can hold tapes for pending instructions before passing into a Turing machine,
  4. An integer to set global bitness (defaults to 32),
  5. A path to the folder containing your Turing machine procedures,
  6. An integer containing the index of a procedure to run next (for jumping),
  7. A separate tape ‚Äúregister‚Äù to store a value to compare other tapes to (for conditional jumps),
  8. An ordered list with class functions to make it operate as a stack,
  9. And an integer to hold the current index of the top of the stack.

With those nine things, it‚Äôs very hard to figure out what you can‚Äôt do, but some additional instructions are necessary.
Now that we‚Äôve reached the point where we are operating on two different classes, I will be using specific words to mitigate confusion. An instruction is a command sent to the Chain() class to perform some tape operation, while a procedure is a file containing transitions for the TuringMachine() class to run. Let‚Äôs start exploring some basic procedures we need to perform common tasks. So far, I‚Äôve only written bitwise procedures, as I am trying to simulate modern computing. Here‚Äôs a list of what I have:

  1. add
  2. and
  3. not
  4. or
  5. xor
  6. tcomp
  7. shl
  8. shr

It should be easy to understand what these procedures do based on their names. The next most important part are the instructions for chaining. Right off the bat, we need a way of sending data to the tape, so I made a put instruction. This is only one of two instructions that actually take input. Currently the put instruction will copy an integer (base 2, 10, or 16) and copy it to the tape in binary, padded with zeros to fill the bitness specified by the class variable. We also need a way of setting and getting tapes to and from the tape registers, respectively. Since we have four registers, we have four store commands (stX, where X is ‚Äòa‚Äô, ‚Äòb‚Äô, ‚Äòc‚Äô, or ‚Äòd‚Äô) and four get commands (gtX, where X is ‚Äòa‚Äô, ‚Äòb‚Äô, ‚Äòc‚Äô, or ‚Äòd‚Äô). I also created a concatenate instruction that will automate the combination of operands separated by a ‚Äò:‚Äô (ctXY, where X and Y are ‚Äòa‚Äô, ‚Äòb‚Äô, ‚Äòc‚Äô, or ‚Äòd‚Äô).
With these instructions, combined with our eight procedures, we can begin writing programs that look an awful lot like Assembly. Here‚Äôs a quick example of one that will perform the previous 2 - 1 example:

put 2
sta
put 1
tcomp
stb
ctab
add

For my chaining implementation, it is important to remember that I tried to make instructions pirform an operation on the input tape. For example, sta will copy the word on the input tape into the A register; put writes a word directly onto the input tape. One instruction that ‚Äúskips‚Äù a step is the ctab one. This instruction will take the tape in the A register and copy it to the input tape, then append a ‚Äò:‚Äô and the tape from the B register (excluding the ‚Äò>‚Äô marker).

Jumping and the stack

I added instructions that will perform a loop backwards through the instruction list. Later, I added ways of pushing and popping the current instruction index to the stack; this allows us to do more advanced jumps and calls, like this:

; code
call 4
call 12
call 19
halt

; 4
    put     1
    sta
    put     2
    stb
    ctab
    add
    sta
    ret

; 12
    put     123456789
    tcomp
    stb
    ctab
    add
    sta
    ret

; 19
    put     987654321
    tcomp
    stb
    ctab
    add
    sta
    ret

call will push the index of the next instruction onto the stack, and set our indexing variable equal to the integer that you pass with the instruction (this is the second of two instructions that take input). ret will pop the top of the stack (make sure you clean it up before this, obviously) and set our indexing variable equal to what is popped. Finally, the halt command will force the chain to stop executing instructions. Without this command, the above example will perform all of its instructions twice: once when they are explicitly called, and again after call 19 returns and the chain continues its top-down execution.

Parting words

Some other things I added was an instruction in the parser that will preload some data from a configuration file. The parser assumes the path to this configuration is the first non-commented line. Here is an example:
