2. Abnormal Entry Point / Section Names

Sometimes compilers create some additional sections in order to store things like debug symbols etc. Whatever I see other then these names instantly raise my suspicion.

5. Data Sections Are Too Large
We generally divide sections into one of two types; code section or data section. As the name suggests code sections are the ones which contains code, and data sections are the ones which contains drumroll data! When a file is packed normally you would only see the functions used by its unpacker stub code. Most probably there will be stuff related to memory allocation, mem permissions, module load, resources operations‚Ä¶ So if it seems like the program imports abnormally few functions you could suspect of a packer.

the name ‚Äúpacker‚Äù is used too broadly nowadays. when they were first introduced their only purpose was to shrink an executable file by using data compression algorithms (like the ones used in winrar, 7zip etc). in those times most antivirus software deployed only static signature based detecion methods. these ‚Äúpackers‚Äù come to corrupt the ‚Äúsignatures‚Äù on an executable while compressing it and hence a packed executable could evade AV detection more easily. little after that this behavior was noticed and these ‚Äúapplication packers‚Äù got widely adopted by malware developers. this is how this innocent tool of binary shrinking got weaponized. then some malware developers rolled out their own packers and also introduced encryption features into these ‚Äúpackers‚Äù, in order to better evade detection. then the term ‚Äúpacker‚Äù evolved into a different and broader meaning than it was before; covering both encryption and compression employment with the purpose of AV evasion. Now they may also contain anti forensics and analysis methods as well. But this is where the term originated.

There are many different types of packers out there. The ones that are concerned with malware and are probably the most common ones come with both encryption and compression. These are labelled ‚Äúrun-time‚Äù (as opposed to ‚Äúscan-time‚Äù) packers meaning that they will deobfuscate the original code in memory and execute it without touching disk. The reason this is important is because if the deobfuscated malware is dropped to the disk (‚Äúscan-time‚Äù), it may trigger an anti-virus to scan its exposed code.

In order to unpack in-memory, the obfuscated malware code is contained within what‚Äôs known as the ‚Äústub‚Äù which is responsible for the recovery and initialisation of the original malware. Essentially, the stub is its own functional code - either shellcode or full PE file - basically a manual executable linker.

Using UPX as an example, UPX compresses the original binary and generates a new PE file. This PE file has at least two main sections, .upx0 and .upx1. upx0 is actually empty in terms of raw ‚Äúdisk‚Äù size but its virtual size (in memory) is as big as the original binary‚Äôs size. upx1 contains both the compressed binary as well as the assembly ‚Äústub‚Äù. What the stub does is it will locate the compressed code and decompress it back into the upx0 in memory (remember the virtual size is big enough to contain the original binary). It will then initialise the original binary by applying relocations(?), rebuilding the import table functions, and then call the .tls functions if they exist. When this is done, it will jump execution to the original binary‚Äôs entry point from where it will continue normal execution like nothing happened.

c0wm1lk:

If they unpack all their instructions right to memory wouldn‚Äôt you just be able to take a copy of the memory and inspect it that way?

Most of the times with the average packer, you can do this. If we are speaking about protectors, it‚Äôs a different story. Parts of the original bytes may be intentionally misplaced or replaced with equivalent code and spread across memory which will provide the same functionality, but results in the original code being fragmented. Some examples of this include API redirection or byte stealing. Protectors like VMProtect may virtualise the code which means that the original code is unrecoverable.

So if you want to get into building these packers, you may need to understand the executable file formats, how they interact with memory, and the Windows operating system.
